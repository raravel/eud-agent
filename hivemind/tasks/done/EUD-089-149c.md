## Context

Live build failure: eudplib's `EUDLoopPlayer("Human")` raised "연결맵에 조건에
맞는 플레이어가 없습니다: 플레이어 종류 Human / 스타트 로케이션을 제대로
배치했는지 확인해보세요" — the test map had no Human slot with a start
location, and the agent had no tool to fix the map's player setup.

User decision (clarify): player-setup scope only (start-location units + OWNR
controllers; no general unit placement), auto-build after apply with the
3-attempt self-fix (EUD-088 covers the build side).

## Implementation

1. **isom-poc CLI** (`MapGenCli.cpp`, the permitted file; solution rebuilt):
   new `playeredit <map.scx> <ops.txt>` subcommand, locedit pattern —
   pipe-separated ops, all-or-nothing (abort BEFORE save), in-place save with
   `autoDefragmentLocations=false` (a player edit must never renumber
   locations). Ops (slots 0-based 0..7, px coords):
   `start|slot|xc|yc` (move the slot's start-location unit 214, or add),
   `delstart|slot` (error when absent),
   `controller|slot|human|computer|rescuable|neutral|inactive|closed`
   (OWNR+IOWN via `setSlotType`, StrScope::Both). Start coords validated
   against map pixel bounds. (isom-poc is not a git repo — file + exe only.)
2. **chk_info.py**: `PLAYER_ACTIONS`/`PLAYER_CONTROLLERS`;
   `MapInfoService.player_setup(action, player=1..8, tile_x, tile_y,
   controller)` — same rails order as location_write (validate → compiling
   guard → lock probe → backup → CLI → re-digest verify returning players[] +
   startLocations[]). Start coords land on the TILE CENTER px.
   `_run_locedit` generalized to `_run_map_edit(subcommand, ..., tool=...)`
   (shared by both subcommands).
3. **tools.py**: `player_setup` ToolSpec (write; required action+player;
   controller enum in the schema so codex sees the legal names);
   `_h_player_setup` validator; `_player_setup_via_service` routing (plan
   gate, budget, journal entry with the backup pointer).
4. **journal.py**: category map + `_map_item` player summaries
   ("P1 controller = human in demo.scx", "P1 start location placed at tile
   (4,8) in demo.scx"); rollback routes player_setup through the same
   restore-backup path; target `player:<action> P<n>`.
5. **engine.py**: `[map locations]` now pins the Human-player requirement and
   the check-then-fix workflow; `[build]` routes the "no matching player"
   failure to player_setup instead of "report it".
6. Docs: features/09 player_setup section, features/05 registry, rules.md map
   rails, architecture.md bullet, isom-poc CLAUDE.md CLI list.
