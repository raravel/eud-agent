## Description
Second live editor E2E defect (EUD-024 panel-creation step, 2026-06-04 23:20, right after EUD-037 unblocked the boot handshake): `createPanel()` fails every Tick with `attempt to call upvalue 'CoreWebView2CreationProperties' (a nil value)` (bridge_error.log, ~1 entry/s). Root cause: the bridge imports the type from the WRONG namespace — `luanet.import_type("Microsoft.Web.WebView2.Core.CoreWebView2CreationProperties")` returns nil because the class lives in the **Wpf** assembly/namespace. Empirically confirmed by reflecting the vendored DLLs: `Microsoft.Web.WebView2.Core.dll` exports NO `*CreationProperties*` type; `Microsoft.Web.WebView2.Wpf.dll` exports `Microsoft.Web.WebView2.Wpf.CoreWebView2CreationProperties` (parameterless ctor + `string UserDataFolder` property). Every other WebView2 API the bridge uses was reflection-verified correct: `Microsoft.Web.WebView2.Wpf.WebView2` (events `CoreWebView2InitializationCompleted`, `NavigationCompleted`; property `CreationProperties: Wpf.CoreWebView2CreationProperties`; property `CoreWebView2: Core.CoreWebView2`; method `EnsureCoreWebView2Async(CoreWebView2Environment)`), `Core.CoreWebView2.Navigate(string)`.

The existing static test (`test_bridge_webview_static.py`) only asserted the loose substring `"CreationProperties"`, which the wrong-namespace import satisfies — it could not catch this.

Fix: change the `import_type` argument to `"Microsoft.Web.WebView2.Wpf.CoreWebView2CreationProperties"` (one line). Strengthen the static test to pin the FULL namespaced type name in the import_type call AND regression-guard that `"Microsoft.Web.WebView2.Core.CoreWebView2CreationProperties"` does not appear anywhere in the bridge.
