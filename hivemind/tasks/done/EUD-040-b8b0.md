## Description
Live editor E2E defect (EUD-024, 2026-06-05): LIST always reports `ftype="?"` for every file, so the server's `_settable_for("?")` returns false for ALL files → every file is treated as read-only → the panel can never select a SET target → the 전송(instruct) button is permanently disabled (and a future settable `.eps` file would also be unselectable). Observed live: `list -> [{"path":"main","ftype":"?","settable":false}]`.

Root cause: the bridge LIST walk reads the file type via `f.Filetype` (lowercase `t`) at `bridge/ZZZ_10_agent_bridge.lua` ~line 409, but the actual VB property name is `FileType` (uppercase `T`). Confirmed by orchestrator LUA reflection on the running editor: `t:GetProperties()` yields `P:FileType:EFileType` (no `Filetype`), and `tostring(f.FileType)` returns `"ClassicTrigger: 6"` (a luanet enum proxy → "Name: value"). The bridge wraps the read in pcall, so the wrong name silently fails to `"?"`. (verify.md e2e step 2 listed this as the EUD-011 deferred `f.Filetype` member confirmation — now resolved: it is `FileType`.)

Fix (bridge LIST handler only):
- Read `f.FileType` (uppercase) instead of `f.Filetype`.
- Normalize the enum proxy string to the bare EFileType NAME so LIST emits `CUIEps` / `ClassicTrigger` / `SCAEps` / ... (not `"CUIEps: 0"`): extract the leading identifier token from `tostring(f.FileType)` with a KopiLua-safe Lua pattern (e.g. `string.match(s, "^%s*([%w_]+)")`). This matches the server contract: `bridge_io._settable_for` does a case-insensitive substring over the family names (`CUI`/`SCA`/`RAWTEXT`), and the panel TargetPicker displays the type — both want the clean name. Keep the existing pcall guard and the `"?"` fallback for any unexpected failure.

After the fix: a `CUIEps`/`SCA*`/`RawText` file resolves to `settable=true` (selectable SET target → instruct enabled); `ClassicTrigger`/GUI stays `settable=false` (correct — read-only). `main` (ClassicTrigger) remains read-only by design; the user must have a CUI/SCA/RawText file to instruct against (separate design discussion).
