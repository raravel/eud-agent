## Description
First-contact editor E2E bug (found live during EUD-024 boot-handshake step, 2026-06-04 22:54, editor running): the bridge spawns the server via the venv launcher `server\.venv\Scripts\python.exe`, which on Windows re-executes the base interpreter as a CHILD process. Observed live: bridge-spawned PID 29472 = venv launcher; real server = PID 34176 (child of 29472, owns the 127.0.0.1:8765 socket). The server writes `"pid": os.getpid()` (the CHILD pid) into server.ready; the bridge's `validateReady()` compares it against `agentProc.Id` (the LAUNCHER pid), mismatches, takes the stale-ready branch, and DELETES server.ready — on every write. `agentSrvReady` never flips, the panel never navigates, while the server itself is healthy (HTTP 200 on `/`, React dist served). Invisible to every headless suite: the runtime rig spawned the server directly (no launcher hop) and the bridge-side check is static-only.

Fix (surgical, both sides of the ownership contract):
- server `app.py`: the server.ready payload adds `"ppid": os.getppid()` next to `pid` (from the child, `os.getppid()` IS the launcher pid the bridge holds). Additive field — existing consumers unaffected.
- bridge `validateReady()`: accept ownership when `ownPid` matches EITHER the ready `pid` OR the ready `ppid` (direct-spawn case keeps matching via `pid`; launcher case matches via `ppid`). Delete the ready file ONLY when NEITHER matches (true foreign/stale file).
- Keep the existing mtime>bridgeStart freshness check unchanged.
