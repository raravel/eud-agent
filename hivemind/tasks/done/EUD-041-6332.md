## Description
Live editor E2E defect (EUD-024, 2026-06-05): the panel shows "RAG 모델 준비 중…" and it never clears, making the user think RAG is stuck. It is NOT stuck — the model warms in ~19s (orchestrator measured cold load 18.7s; GPU load confirmed). The display is the bug.

Root cause: the server's RAG warmup reports progress as `{type:"progress", stage:"rag_warmup", detail:"started"}` then `{... detail:"done"}` (or `detail:"error: ..."`) — see `server/eud_agent/app.py` `_warmup_progress` (~line 407) and `rag.start_warmup`. But the panel's `App.tsx` `onMessage` progress case (~line 67-69) labels purely by `stage` via `STAGE_LABELS` and IGNORES `msg.detail`, so BOTH `started` and `done` render the same "RAG 모델 준비 중…" log line — completion is never shown. (This warmup progress is a boot-time broadcast; it does not change the store phase, so there is no spinner — just a stale text line.)

Fix (panel only): in `App.tsx` `onMessage`, handle the `rag_warmup` stage's `detail`:
- `started` → "RAG 모델 준비 중…" (progress kind, as today)
- `done` → "RAG 모델 준비 완료" (ok/info kind) — or suppress, but a one-line completion is clearer
- `error: <detail>` → "RAG 사용 불가: <detail>" (warn kind)
Other stages (`rag`, `codex`, `lsp`, `waiting_build`) are instruct-flow progress and keep current behavior (they carry no meaningful detail for labeling, and `progressReceived(stage)` phase handling — only `waiting_build` flips phase — must stay unchanged). To make this unit-testable, extract the progress→log mapping into a small pure helper (e.g. `progressLabel(stage, detail) -> {kind, text}` in a lib module) and have `onMessage` call it; the store interaction (`progressReceived`) stays as-is.

Note: the WS protocol type for `progress` must carry the optional `detail` field if it does not already — extend `panel/src/ws/protocol.ts`'s progress message type to include `detail?: string` (the server already sends it). Verify the existing protocol type and add `detail` if missing (in scope).
