## Description
Speed up and cheapen the `build-rag-index` CI job by migrating it from `windows-latest` to
`ubuntu-latest` and caching the fastembed bge-m3 model (~570MB) across runs. Today the model
is re-downloaded every run (only cargo is cached via `Swatinem/rust-cache`) and Windows
minutes bill at 2x. The builder already accepts `--cache <dir>` to relocate the fastembed
model cache (`ci/build_rag_index.rs`), so the workflow just needs to point at a cached dir
and restore/save it.

Required changes (edit ONLY `.github/workflows/build-rag-index.yml`):
- Switch `runs-on: windows-latest` to `runs-on: ubuntu-latest`.
- Port the "Generate release manifest" step from PowerShell to bash (version resolution,
  sha256 read from the sidecar, manifest JSON written UTF-8 without BOM, `GITHUB_OUTPUT`).
- Drop the `.exe` suffix and backslashes in the builder path:
  `./ci/target/release/build_rag_index --corpus ci/corpus --cache "$RUNNER_TEMP/fastembed" --out rag-index.bin`.
- Add an `actions/cache` step keyed on the model id (e.g. `bge-m3`) that restores/saves the
  fastembed cache dir passed via `--cache`, so the 570MB download is reused across runs.
- Keep the triggers (`workflow_dispatch`, `rag-index-v*` tag, `push` on `main`+`ci/corpus/**`),
  the `Swatinem/rust-cache` cargo cache, and the `softprops/action-gh-release@v2` publish step.
