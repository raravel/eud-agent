# Zombie server leak: superseded server never self-terminates on quick editor restart

## Symptom (live, 2026-06-05)

Editor restarted at 10:22; the previous session's server pair (launcher 29884 +
child 31944, parent = dead editor 34160) survived indefinitely with bge-m3
loaded (GPU memory held). Confirmed live: two server pairs polling the same
`Data\agent\{inbox,outbox}` simultaneously.

## Root cause

The server's ONLY shutdown path is heartbeat staleness (>60s on
`Data\agent\heartbeat.txt`). On a quick editor restart the NEW editor instance
resumes writing the SAME heartbeat file, so the old server never observes
staleness and never exits. Zombies accumulate per restart; each holds GPU
memory and races the new server for IPC `.result` files (both poll `srv-*`
basenames in the shared outbox).

## Fix (supersede check)

In the existing 15s heartbeat-watch loop in `server/eud_agent/app.py`, also
read `Data\agent\server.ready`:

- If the file exists, parses as JSON, and its `token` differs from this
  process's own token -> a newer server owns the data dir -> log and
  self-terminate.
- On THIS exit path, do NOT delete `server.ready` (it belongs to the new
  server). The normal graceful-shutdown deletion must be skipped when exiting
  because superseded.
- If the file is missing or unparsable -> no decision (transient states: bridge
  deletes stale ready at init before the new server writes; corrupt/partial
  reads). The heartbeat rule remains the fallback.

Token comparison (not pid) is authoritative: the server knows the exact token
it wrote; pid comparison is ambiguous (launcher vs child, EUD-037).

## Acceptance

- Unit tests (tmp data dir, watcher logic exercised directly or with short
  intervals):
  1. ready exists with a DIFFERENT token -> server terminates AND
     `server.ready` is NOT deleted by the dying server.
  2. ready exists with the OWN token -> no exit.
  3. ready missing -> no exit (heartbeat staleness still the only rule).
  4. ready corrupt/unparsable -> no exit.
- Existing heartbeat-staleness self-termination behavior unchanged (existing
  tests stay green).
- Full suite green.

## Constraints

- IPC/state files UTF-8 without BOM (utf-8-sig forbidden).
- Do not change the bridge; this is server-only.
- Keep `server.ready` atomic-write semantics untouched.

## Threshold decision (2026-06-05)

Keep the 60s heartbeat-staleness threshold unchanged (user decision): the measured 54s UI-thread stall (EUD-039) proves shorter thresholds cause spurious self-termination. Quick-restart zombies are handled by THIS task's supersede check (<=15s), not by lowering the staleness window.
