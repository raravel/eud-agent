## Bug report (user, live panel observation — SERVER half)

codex reasoning is invisible and answer/delta text never reaches the panel. SDK ground truth (verified in the pinned `openai_codex` 0.1.0b3 source — `generated/notification_registry.py:99-101`, `v2_all.py:92-99, 2804-2823`):

- `item/reasoning/summaryTextDelta` and `item/reasoning/textDelta` notifications carry the reasoning text in payload field `delta: str` — `_classify_event` in `agent_runner.py` has no branch for them; they fall through to `("event", method)` and the text is dropped.
- `item/agentMessage/delta` (AgentMessageDeltaNotification, payload `delta: str`) is mapped to `("delta", "")` — the answer text is discarded.

The PANEL half (rendering reasoning dim, answer prominent, no raw kind leakage) is EUD-065's AI Elements + Streamdown rework. This task is the server contract it consumes (features/05 "WS protocol v2", amended 2026-06-05).

## Fix design

`agent_runner.py` `_classify_event`:
- `item/reasoning/summaryTextDelta` + `item/reasoning/textDelta` → `("reasoning", <payload.delta>, info)`.
- `item/agentMessage/delta` → `("delta", <payload.delta>, info)`.
- Payload access stays defensive (`getattr`, `or ""`) per the module convention — a missing field degrades to empty detail, never crashes the turn loop.
- `item/completed` with `rtype == "reasoning"` keeps its current generic mapping (the panel filters); no other kind changes.

## Completion criteria

- [ ] `_classify_event` maps both reasoning delta methods to kind `reasoning` with the delta text as detail (unit tests with fake notification objects: summaryTextDelta, textDelta, and a payload missing `delta` → empty detail, no raise).
- [ ] `_classify_event` carries `payload.delta` text on `agentMessage/delta` events (test).
- [ ] Existing kinds (thinking/tool_call/tool_result/turn_done/answer/token_usage) unchanged (regression assert).
- [ ] Server suite (pytest) + ruff green.
