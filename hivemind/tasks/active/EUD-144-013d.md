## Description
Reduce embedding wall-time in `ci/build_rag_index.rs` by tuning batch size and ONNX Runtime
thread usage. Today `embed(&texts, None)` uses the default batch and ORT picks its own thread
count. Pass an explicit batch size and configure ORT intra-op threads to the available cores
so the ~8445-chunk embedding step utilizes the runner fully. This must NOT change the index
contents semantically — batching only affects throughput, not per-input embeddings.

Required changes (edit ONLY `ci/build_rag_index.rs`):
- Pass an explicit batch size to `embedder.embed(&texts, Some(<batch>))` (tune empirically;
  a larger batch amortizes per-call overhead). Make it overridable via an env var or `--batch`
  flag with a sensible default.
- Configure ORT intra-op thread count to the host core count (e.g. via `Bgem3InitOptions`
  thread option if exposed, otherwise the appropriate ORT/`OMP_NUM_THREADS` env honored by
  ort 2.0.0-rc.12) so embedding is not single-threaded.
- Keep output bytes equivalent to the untuned build for the same corpus (batching is
  deterministic per input); the index must still pass the existing build + any rag parity check.
