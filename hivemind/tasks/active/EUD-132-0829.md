## Description
Wire the first-run bootstrap into app startup. `bootstrap.rs` (EUD-100) is code-complete
(asset_status / verify_and_place / ensure_model / ensure_rag_index, sha256 + atomic place) but is
NEVER called — `lib.rs` only declares `pub mod bootstrap;`. The app currently starts without
ensuring the model/RAG-index assets exist.

- In the Tauri `setup` hook, resolve data dirs (config.rs `DataDirs`), check the manifest, and run
  the bootstrap flow (ensure RAG index from GitHub Release, ensure bge-m3 model) on a background
  task so the panel stays usable; emit `progress {stage: bootstrap}` (and the existing rag_warmup)
  to the panel. Missing/corrupt asset re-downloads; readiness is NOT gated on model load (lazy +
  background warmup per rules.md).
- Do not block the window; surface download progress/errors to the panel (consumed by EUD-120 UI).
- SCOPE EXPANDED 2026-06-11 (user-reported scenario gap): the feature-10 manifest check is
  3-part — **editor-path config** + model + RAG index — and the editor-path leg had no flow.
  Added: `setup.rs` with `setup_status` (manifest snapshot), `setup_pick_editor_path` (native
  folder picker -> `validate_editor_path` -> `save_config`), `bootstrap_run` (panel-driven
  download/retry; the RAG spec resolves from the published `rag-index.manifest.json` release
  asset); SetupScreen gained the pick step; first run is panel-driven (pick THEN download),
  auto-bootstrap on later launches only. `bridge_from_config` with an empty editor path now
  fails with "editor path not configured" (distinct from the stale-heartbeat
  "editor not connected") so the panel routes to setup instead of the reconnect notice.
