Diagnosis: the two build attempts are mutually exclusive, proving a single-CRT-per-binary
conflict. `isom_capi.lib` (and its folded IsomTerrain/CascLib/StormLib/ICU/MappingCore
archives) is built `/MT` (static CRT) + `/GL` (WPO) per `native/isom/**` vcxproj

`/NODEFAULTLIB:msvcrt.lib` + `/DEFAULTLIB:libcmt.lib`. But `ort_sys` (the prebuilt ONNX
runtime pulled in by `fastembed`, already an `eud-agent` dependency) requires `/MD` (dynamic
CRT). eud-agent links **both** isom and ort for the first time in this task, so the conflict
surfaces now. `isom-sys`'s `/NODEFAULTLIB` link-args are also non-transitive (they don't
reach the `eud-agent` final link), which is why the default build leaves the isom shim
symbols unresolved.

## Incident

### What broke
- `cargo build -p eud-agent` and `cargo test -p eud-agent mapsafe` cannot LINK. The
  completion criteria "`cargo build -p eud-agent` links" and "`cargo test -p eud-agent
  mapsafe` passes" are unsatisfiable within the declared scope.

### Why
- CRT-uniformity conflict: `isom_capi.lib` is `/MT`+`/GL` (static CRT, set in
  `native/isom/**` vcxproj `RuntimeLibrary` + `WholeProgramOptimization`); `ort_sys`
  (fastembed/RAG) is `/MD` (dynamic CRT). A single MSVC binary can use only one CRT, and
  this task is the first to link isom and ort together. Reconciling them requires rebuilding
  the vendored engine as `/MD` (vcxproj `RuntimeLibrary` -> `MultiThreadedDLL` /
  `MultiThreadedDebugDLL`, drop `/GL`) and removing the static-CRT forcing in
  `crates/isom-sys/build.rs` — both **outside** `src-tauri/**`.

### What fixed it
- Not fixed. Blocked by user decision (block + re-plan). The CRT reconciliation should be a
  separate planned task scoped to `native/isom/**` + `crates/isom-sys/build.rs`; after it
  lands, retry EUD-128 (the preserved branch already has the correct src-tauri code).
